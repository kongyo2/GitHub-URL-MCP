# 開発ワークフロー

MCP サーバー開発ワークフロー (FastMCP ボイラープレート使用)
このドキュメントは、FastMCP ボイラープレート を基盤として、LLM エージェントが利用する MCP (Model Context Protocol) サーバーを構築・開発するための標準的なワークフローを説明します。

## ステップ

1. 現在のディレクトリを把握します。
2. 依存関係のインストール: npm install を実行して、必要な Node.js パッケージをインストールします。
3. package.jsonを編集して、プロジェクト固有の情報に更新
4. 厳密にボイラープレートを遵守して機能の実装を実行する。テンプレートに背いて、実装を怠けたいが為に全てをserver.tsに完結させる等の勝手な真似はしないこと。
5. 引き続きボイラープレートを遵守して実装したツールそれぞれに対応するテストを実装します。実装後、必ずsequentialthinking を呼び出して、テストがより完璧な実装であるかをultrathinkで批判的に見直し、より完璧なテストの実装を行います。この手順を飛ばして勝手に進まないようにして下さい。
6. npm run testで何度でもテストを実行し、徹底的なデバッグと修正を行います。
7. テストが全て通った後、 npm run lint と npm run formatを実行し、エラーが一つも出ないように修正します。
8. 再度npm run testで何度でもテストを実行し、徹底的なデバッグと修正を行います。
9. 最後にnpm run build を実行し、エラーがあった場合は適切な修正手順に戻ります。
10. 完了報告をユーザーに行います。

# githubリポジトリ参照ワークフロー

任意のリポジトリのコードを参照するためのワークフローを説明します。

## ステップ

1. 参照するように頼まれたリポジトリのオーナー名とリポジトリ名を把握します。
2. MCP Docs の fetch_generic_documentationを先程把握したオーナー名とリポジトリ名で呼び出します。 当該ツールの説明にも明記されているように、オーナー名とリポジトリ名を把握しているにもかかわらずmatch_common_libs_owner_repo_mappingを代わりに呼び出して見付からなかったという言い訳を予めしようとする行為はユーザーに対する怠慢であり不誠実です。また、無理と間違った名前でツールを呼び出した上でわざと呼び出しを失敗させて手順を省略する行為も同様に忌むべき行為です。
3. search_generic_code を必ず省略せずに実行し、fetch_generic_url_contentで必要なコードを全て参照します。変なqueryでsearch_generic_codeを実行した上で結果が帰って来ないという見え透いた手順を怠ける試みはユーザーの強制停止によって阻止されます。
4. 必要な全ての**コードを**参照したことを確認した上でMCP Docsでのリポジトリの参照を完了します。なお、**ドキュメント**をのみを参照してお茶を濁す行為は厳禁です。
5. 次にdeepwikiを同じリポジトリで呼び出して全てのコンテンツを読み込みます。その後deepwikiのask_questionでここまでで分からない点を必ず質問します。この際の質問は必ず細かい要件を明示的にした上で、十分な長文で質問して下さい。
